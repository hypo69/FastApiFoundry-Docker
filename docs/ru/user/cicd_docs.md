# CI/CD и автоматизация разработки

Данный раздел описывает механизмы автоматизации контроля качества, управления версиями и процесса выпуска релизов в проекте AI Assistant.

## 1. GitHub Actions

Проект использует два основных конвейера (workflows) для обеспечения стабильности:

### QA Pipeline (`dev-qa.yml`)
Запускается автоматически при push или PR в ветку `dev`.
*   **Что делает:** Устанавливает зависимости, выполняет линтинг Dockerfile, аудит безопасности (`pip-audit`), запускает `Invoke-Qa.ps1` и обновляет бейдж покрытия кода (coverage badge) в документации.
*   **Безопасность:** Если обнаружены критические уязвимости в зависимостях, workflow автоматически создает Issue в GitHub с детальным отчетом.

#### Настройка Telegram уведомлений
Для получения отчетов о работе пайплайна в Telegram необходимо настроить **GitHub Secrets**:
1. Перейдите в `Settings` -> `Secrets and variables` -> `Actions` вашего репозитория.
2. Нажмите `New repository secret` и добавьте:
    - **`TELEGRAM_TOKEN`**: API токен бота, полученный от @BotFather.
    - **`TELEGRAM_TO`**: Ваш числовой Telegram ID (или ID группы).

Пайплайн использует эти данные для отправки статуса сборки и ссылок на найденные уязвимости.

### Release Pipeline (`release.yml`)
Запускается при создании Git-тега в формате `v*.*.*`.
*   **Что делает:** Собирает проект на Windows-агенте, прогоняет финальные тесты и создает ZIP-архив с релизной версией.
*   **Артефакты:** Готовый архив доступен в разделе Artifacts соответствующего запуска.

## 2. Локальные скрипты деплоя

В директории `scripts/` сосредоточены инструменты для ручного управления циклом выпуска:

*   **`New-Release.ps1`**: Интерактивный помощник для повышения версии (Major.Minor.Patch). Он обновляет файл `VERSION`, создает коммит и проставляет локальный Git-тег.
*   **`Build-Release.ps1`**: Скрипт «чистой» сборки. Он удаляет временные файлы (`__pycache__`, логи, venv), запускает `Invoke-Qa.ps1` и упаковывает всё содержимое в ZIP-архив в папку `release/`.
*   **`Invoke-EnvMigration.ps1`**: Помогает при обновлении. Сравнивает ваш `.env` с актуальным `.env.example` и подсказывает, какие новые секреты или настройки нужно добавить.

## 3. Git Hooks

Для предотвращения попадания "сломанного" или небезопасного кода в репозиторий используются Git Hooks.

### Установка
Хуки настраиваются автоматически при первом запуске `Invoke-Qa.ps1` (через вызов `Setup-GitHooks.ps1`).

### Реализованные проверки
*   **Pre-commit Hook**: Перед каждым коммитом запускается быстрая проверка линтером (Ruff). Если найдены критические ошибки оформления, коммит будет отклонен до их исправления.
*   **Учетные данные**: Интегрированный `detect-secrets` проверяет, не пытаетесь ли вы случайно закоммитить API-ключи или пароли.

## 4. Жизненный цикл релиза (Workflow)

Рекомендуемый порядок действий для выпуска новой версии:

1.  Завершить разработку функционала.
2.  Запустить `scripts/Invoke-Qa.ps1` и убедиться, что все тесты (`Pytest`), типы (`Mypy`) и стиль (`Ruff`) в порядке.
3.  Запустить `scripts/New-Release.ps1`, выбрать тип обновления и подтвердить создание тега.
4.  Выполнить `git push --tags`.
5.  Дождаться завершения **Release Pipeline** на GitHub.
6.  Скачать готовый архив и использовать `Invoke-EnvMigration.ps1` для обновления настроек на целевой машине.
