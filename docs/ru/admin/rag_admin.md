# RAG — управление индексами

Администратор управляет RAG-профилями: создаёт, загружает документы, переключает активный профиль.

---

### Почему RAG — это сложная инженерная система в проекте

В отличие от простых реализаций RAG «из коробки», архитектура поиска в этом проекте построена на принципах максимальной автономности, масштабируемости и гибкости:

1.  **«Всеядность» и глубокое извлечение (Universal Extraction)**:
    Система не просто читает файлы. `TextExtractor` — это сложный конвейер, который умеет рекурсивно обрабатывать архивы (до 3 уровней вложенности), запускать OCR для изображений через Tesseract и транскрибировать аудио через локальный Whisper. Это превращает любую «свалку» документов в структурированную базу знаний.

2.  **Гибридное хранилище (FAISS + SQLite)**:
    Для обеспечения скорости и удобства управления используется связка двух технологий. **FAISS** (Facebook AI Similarity Search) отвечает за мгновенный векторный поиск, а **SQLite** (`documents.db`) хранит метаданные, исходные тексты и связи. Это позволяет не только искать, но и точечно редактировать или удалять документы без полной пересборки индекса.

3.  **Профильная изоляция (Hot-swappable Profiles)**:
    RAG в проекте — это не одна монолитная база. Архитектура поддерживает неограниченное количество изолированных профилей (например: «Юристы», «Техподдержка», «Личные заметки»). Переключение между ними происходит «на лету» через API, что позволяет модели менять контекст за миллисекунды без перезагрузки сервера.

4.  **Инкрементальное управление (Lifecycle Management)**:
    Реализована логика «умного» обновления. Вы можете добавить один файл в базу из 100 000 документов, и система не будет пересчитывать всё. Механизм **Compact** позволяет проводить «санитарную очистку» векторного пространства, удаляя неактуальные векторы и оптимизируя использование оперативной памяти.

5.  **Полная локальность эмбеддингов (Privacy by Design)**:
    В отличие от многих решений, использующих внешние API (например, OpenAI Embeddings), здесь векторы генерируются библиотекой `sentence-transformers` прямо на CPU/GPU пользователя. Это гарантирует, что даже на этапе индексации ни один фрагмент ваших данных не покидает локальный контур.

**Резюме**: RAG-система проекта — это полноценная локальная поисковая машина. Она превращает стандартную LLM в эксперта по вашим частным данным, сохраняя при этом высокую производительность и стопроцентную приватность.

## Глубокое погружение в механизмы RAG

### Связь между SQLite и FAISS

В RAG-системе **SQLite (`documents.db`)** и **FAISS (`faiss.index`)** работают в тандеме, выполняя разные, но взаимодополняющие функции:

-   **FAISS (Facebook AI Similarity Search)**: Это высокооптимизированная библиотека для эффективного поиска ближайших соседей в векторном пространстве. Она хранит только числовые векторы (эмбеддинги) и их уникальные идентификаторы (ID). FAISS невероятно быстр для поиска, но не предназначен для хранения текстового контента, метаданных или для легкого удаления отдельных векторов. Удаление вектора из плотной структуры FAISS — дорогая операция.
-   **SQLite (`documents.db`)**: Это легковесная реляционная база данных, которая хранит всю "человекочитаемую" информацию:
    -   **Текстовые чанки**: Фактический текст, извлеченный из документов и разбитый на фрагменты.
    -   **Метаданные**: Путь к исходному файлу, дата индексации, профиль RAG и, самое главное, флаг `is_active`.
    -   **Уникальные ID чанков**: Каждый текстовый чанк в SQLite имеет уникальный идентификатор, который **точно соответствует ID вектора в индексе FAISS**.

**Как они связаны:**
Когда документ индексируется, каждый его текстовый чанк получает уникальный ID. Этот же ID используется для соответствующего вектора, который добавляется в FAISS. Таким образом, SQLite хранит "что" (текст и метаданные), а FAISS хранит "где" (векторное представление для поиска) и "как" (ID для связи).

**При удалении документа (Soft-Delete):**
Вместо того чтобы пытаться удалить векторы из FAISS (что неэффективно), система просто помечает соответствующие чанки в SQLite как `is_active = 0`. Векторы в FAISS остаются, но при поиске и отображении результатов система фильтрует только те чанки, у которых `is_active = 1`. Это позволяет быстро "удалять" документы без перестройки всего индекса.

**При поиске:**
1.  Пользовательский запрос векторизуется.
2.  FAISS быстро находит N ближайших векторов (и их ID) к вектору запроса.
3.  Эти ID используются для выборки соответствующих текстовых чанков и их метаданных из SQLite.
4.  Система фильтрует результаты, показывая только активные чанки (`is_active = 1`).

### Механизм Compact: Очистка и оптимизация

В системе используется архитектура **Soft-Delete**. При удалении документа записи в SQLite помечаются как неактивные, но векторы в FAISS остаются. Механизм **Compact** необходим для физической очистки:

1.  **Инициирование**: Когда количество «мертвых» векторов превышает 20% или по требованию администратора.
2.  **Реконструкция**: Система считывает все чанки из SQLite, у которых флаг `is_active = 1`.
3.  **Пересборка**: Создается новый индекс FAISS, в который заново добавляются только актуальные векторы.
4.  **Атомарная замена**: Старый файл `faiss.index` заменяется новым.
5.  **Результат**: Уменьшение потребления RAM и ускорение поиска за счет устранения фрагментации.

**Автоматический запуск Compact**:
Для автоматизации `Compact` система может периодически (например, раз в 24 часа, настраивается через `rag_system.compact_check_interval_hours` в `config.json`) проверять процент неактивных чанков. Если он превышает `compact_threshold_percent`, операция `Compact` запускается как **фоновая асинхронная задача**, чтобы не блокировать основной поток API.

### Расширение TextExtractor

`TextExtractor` использует конвейерную обработку. Чтобы добавить поддержку нового формата файлов, необходимо:

1.  **Добавить расширение**: В классе `TextExtractor` обновить список `SUPPORTED_EXTENSIONS`.
2.  **Создать обработчик**: Реализовать метод `_extract_format_name`, который возвращает строку текста.
3.  **Зарегистрировать маппинг**: В методе `extract_text` добавить условие, связывающее расширение файла с новым обработчиком.

*Пример добавления поддержки `.log` файлов:*
```python
def _extract_log(self, file_path: Path) -> str:
    with open(file_path, 'r', encoding='utf-8', errors='ignore') as f:
        return f.read()
```
*Примечание: Если формат сложный, рекомендуется использовать интеграцию с библиотекой `markitdown` или специализированными парсерами.*

## Управление RAG через API

### Получение статистики (Check Stats)
**Эндпоинт:** `GET /api/v1/rag/stats`
**Пример ответа:**
```json
{
  "profile_name": "legal_docs",
  "total_chunks": 5000,
  "active_chunks": 4800,
  "inactive_chunks": 200
}
```

Возвращает детальную информацию о текущем индексе:
- `total_chunks`: Общее количество векторов в FAISS.
- `active_chunks`: Количество векторов, помеченных как активные в SQLite.
- `inactive_chunks`: Количество "мертвых" векторов (мусора).

*Пример расчета процента мусора:* `(inactive_chunks / total_chunks) * 100`.

### Принудительный Compact
**Эндпоинт:** `POST /api/v1/rag/compact`

Запускает процесс пересборки индекса. Рекомендуется вызывать, если процент неактивных записей выше 20-30%.

## Обслуживание и мониторинг

### Проверка доступности перед операциями
Перед запуском тяжелых операций (таких как `Compact` или `Build Index`), рекомендуется проверять статус сервера.

**Пример на PowerShell:**
```powershell
try {
    $check = Invoke-RestMethod -Uri "http://localhost:9696/api/v1/health" -Method Get -TimeoutSec 5
    if ($check.status -eq "healthy") { 
        # запуск операции
    }
} catch {
    Write-Error "Сервер недоступен"
}
```

### Работа с несколькими профилями
Поскольку API возвращает статистику только по **активному** профилю, для мониторинга всей системы (всех профилей RAG) используйте следующую стратегию:

1. Вызовите `GET /api/v1/rag/profiles` для получения списка всех имен.
2. Перебирайте имена профилей.
3. Для детального анализа каждого профиля используйте переключение: `POST /api/v1/rag/profiles/{name}/activate`.

!!! note "Рекомендация для разработчика"
    Если вам часто требуется проверять количество неактивных документов во всех базах, вы можете реализовать прямой запрос к SQLite файлам `documents.db` в каждой подпапке директории RAG. Это работает значительно быстрее, чем последовательная активация профилей через API, так как не требует загрузки векторов в оперативную память.

### Статусы очистки
| Состояние | Действие |
|---|---|
| Неактивных чанков < 10% | Очистка не требуется. |
| Неактивных чанков 10-25% | Рекомендуется плановая очистка (`Compact`). |
| Неактивных чанков > 25% | Требуется немедленная очистка для восстановления производительности. |

## Автоматизация обслуживания

### Запуск Compact по расписанию (PowerShell)

Для поддержания индекса в чистоте без участия пользователя можно создать скрипт и добавить его в Планировщик задач Windows.

1. **Скрипт**: Вызывает `GET /api/v1/rag/stats`, вычисляет процент мусора и при необходимости вызывает `POST /api/v1/rag/compact`.
2. **Планировщик**: Рекомендуется запускать проверку раз в сутки или раз в неделю в часы низкой нагрузки.

```powershell
# Пример команды для регистрации задачи
$action = New-ScheduledTaskAction -Execute "powershell.exe" -Argument "-File C:\scripts\compact_rag.ps1"
$trigger = New-ScheduledTaskTrigger -Daily -At 3am
Register-ScheduledTask -TaskName "RAG-Maintenance" -Action $action -Trigger $trigger
```

!!! info "Встроенная автоматизация"
    Система также поддерживает автоматический запуск на основе параметров `compact_threshold_percent` и `compact_check_interval_hours` в `config.json`. Внешний скрипт PowerShell рекомендуется для более гибкого управления временем запуска (например, только ночью).

## Справочник параметров конфигурации

Настройки RAG системы в `config.json` (секция `rag_system`):

| Параметр | Описание | Значение по умолчанию |
|---|---|---|
| `compact_threshold_percent` | Процент "мусорных" векторов, при котором запускается Compact | `20` |
| `compact_check_interval_hours` | Периодичность автоматической проверки состояния индекса | `24` |

## Где хранятся индексы

```
~/.ai-assist/rag/
├── train_data/           ← активный профиль (задан в config.json)
│   ├── faiss.index       ← FAISS IndexIDMap
│   ├── documents.db      ← SQLite: документы и чанки
│   ├── meta.json         ← chunks, model, updated_at
│   └── README.md         ← описание (отображается в UI)
├── project_docs/
│   └── ...
└── legal/
    └── ...
```

Активный профиль задаётся в `config.json`:

```json
{
  "rag_system": {
    "index_dir": "~/.ai-assist/rag/train_data",
    "enabled": true
  }
}
```

## Загрузка документов через веб-интерфейс

Вкладка **RAG → Index Documents** содержит две вкладки.

### ZIP / Files

Загрузка одного или нескольких файлов/архивов из браузера.

- Поддерживаемые форматы: `zip`, `7z`, `rar`, `tar`, `tar.gz`, `tgz`, `pdf`, `docx`, `md`, `txt`, `html`, `xlsx`, `pptx`
- Содержимое архивов рекурсивно извлекается и индексируется как единый документ
- Прогресс отображается пофайлово: extract → chunk → embed → index
- Endpoint: `POST /api/v1/rag/index/stream` (SSE)

### Directory

Загрузка всех файлов из локальной папки через нативный диалог браузера.

- Используется `<input webkitdirectory>` — браузер открывает системный диалог выбора папки
- Все файлы внутри папки (включая подпапки) передаются на сервер
- Каждый файл индексируется отдельно через тот же SSE endpoint
- Относительный путь (`ru/about.md`) сохраняется как `source_name` в метаданных

!!! warning "Временные файлы при загрузке"
    Все загружаемые файлы временно сохраняются на диск сервера перед обработкой.
    Это архитектурное ограничение: MarkItDown, zipfile, tarfile, py7zr и rarfile
    работают с путями к файлам, а не с байтами в памяти.
    Временные файлы создаются через `tempfile.mkstemp()` в %TEMP% или /tmp
    и удаляются сразу после обработки.

!!! info "Имена файлов с путями (webkitdirectory)"
    При загрузке директории браузер передаёт `filename` как относительный путь
    (например `ru/about.md`). Сервер использует его только как метку источника
    в индексе — для создания временного файла берётся только расширение из `basename`.

## Построить индекс из серверной директории

Если документы уже находятся на сервере — используйте `POST /api/v1/rag/build`:

```bash
POST /api/v1/rag/build
{
  "docs_dir": "C:/Documents/MyProject",
  "model": "sentence-transformers/all-mpnet-base-v2",
  "chunk_size": 1000,
  "overlap": 50,
  "force": false
}
```

Ответ:

```json
{
  "success": true,
  "chunks": 342,
  "rebuilt": true,
  "index_dir": "~/.ai-assist/rag/MyProject",
  "message": "Index rebuilt"
}
```

## Переключить активный профиль

```bash
# Активировать профиль
POST /api/v1/rag/profiles/project_docs/activate

# Отключить RAG (без удаления индексов)
POST /api/v1/rag/profiles/deactivate
```

В веб-интерфейсе: **RAG → RAG Bases** → кнопка ▶ рядом с профилем.

## Document Manager

Вкладка **RAG → Document Manager** позволяет управлять документами инкрементально:

- Добавить документ вручную (текст + заголовок)
- Редактировать — переиндексируются только изменённые чанки
- Удалить — soft-delete, векторы остаются в FAISS до Compact
- **Compact** — перестроить FAISS из активных чанков (появляется когда >20% неактивных)

## README.md для профиля

Каждый профиль может содержать `README.md` с описанием:

```markdown
# Документация проекта

Техническая документация по проекту XYZ.
Содержит: архитектурные решения, API описания, инструкции по развёртыванию.
```

API возвращает это описание при `GET /api/v1/rag/profiles`.

## Миграция legacy-индекса

Старые профили с `chunks.json` (без SQLite) нужно мигрировать:

```bash
POST /api/v1/rag/migrate/index-id-map
```

Миграция импортирует чанки в `documents.db` и перестраивает `faiss.index` как `IndexIDMap`.
