# Безопасность и изоляция данных в архитектуре MCP

## Введение

Одним из главных барьеров при внедрении AI в корпоративную среду является риск несанкционированного доступа модели к чувствительным данным. Архитектура **Model Context Protocol (MCP)** в нашем проекте решает эту задачу через физическую и логическую изоляцию инструментов от самой модели.

## Как обеспечивается изоляция?

### 1. Процессная изоляция (Sandboxing)
Каждый MCP-сервер запускается как **отдельный операционный процесс**. Это означает, что:
- У сервера есть собственные ограничения по памяти и правам доступа.
- Сбой в одном сервере не влияет на работу всей системы.
- Языковая модель не имеет прямого доступа к файловой системе или сети; она может лишь отправлять запросы серверу через строго определенный интерфейс.

### 2. Принцип минимальных привилегий
В отличие от монолитных агентов, имеющих «права администратора» на всё, MCP-серверы предоставляют только те функции, которые им разрешены:
- **Узкая специализация:** Сервер для работы с SQL-базой видит только эту базу и только разрешенные таблицы.
- **Изоляция секретов:** Ключи API и пароли хранятся в переменных окружения конкретного MCP-сервера. Сама LLM-модель никогда не видит эти учетные данные — она лишь просит сервер «выполни действие X», и сервер делает это, используя свои секреты.

### 3. Прозрачность и аудит (Audit Trail)
Все взаимодействия между AI-агентом и MCP-сервером проходят через наш оркестратор. Это позволяет:
- Логировать каждый вызов инструмента.
- Видеть, какие именно данные были переданы на вход и какой результат получен.
- Внедрять системы подтверждения (Human-in-the-loop) для критических операций (например, перед выполнением `DROP TABLE` или отправкой письма).

### 4. Отсутствие общего контекста данных
MCP-серверы не делятся данными друг с другом напрямую. Если агенту нужно объединить данные из Jira и из базы данных, он должен получить их по отдельности. Это предотвращает случайное «смешивание» конфиденциальной информации из разных источников внутри одного процесса обработки.

## Преимущества для безопасности компании

| Угроза | Решение через MCP |
| :--- | :--- |
| **Инъекция промптов** | Даже если злоумышленник заставит модель попытаться прочитать пароли, у модели нет технического доступа к файлам — только к инструментам MCP. |
| **Утечка данных в облако** | Все данные обрабатываются локальными MCP-серверами. В облако (если оно используется) уходит только финальный результат, а не исходные дампы баз. |
| **Неконтролируемые действия** | Администратор может в любой момент отключить конкретный MCP-сервер, моментально лишив AI доступа к определенной системе без перенастройки всей модели. |

## Заключение

Использование MCP превращает AI из «черного ящика» с неясными полномочиями в контролируемый набор инструментов. Это позволяет ИТ-департаментам гибко настраивать доступы и гарантировать, что искусственный интеллект работает только в рамках дозволенных границ.
