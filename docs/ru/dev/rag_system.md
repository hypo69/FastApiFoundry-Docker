# Система RAG (Retrieval-Augmented Generation)

Модуль поиска и извлечения контекста (Retrieval-Augmented Generation) в FastAPI Foundry предназначен для обогащения ответов ИИ-моделей релевантной информацией из ваших документов. Это позволяет моделям давать более точные, актуальные и обоснованные ответы, используя ваши собственные знания.

## 🗺️ Архитектура RAG

Система RAG работает по следующему принципу:

```
Документы (docs/, .md, .txt, .html)
        ↓
  rag_indexer.py          ← (Запускается вручную один раз для построения индекса)
        ↓
  rag_index/
    ├── faiss.index       ← Векторный индекс FAISS
    ├── chunks.json       ← Текст и метаданные чанков
    └── index_info.json   ← Модель, размерность, дата создания индекса
        ↓
  rag_system.py           ← Загружает индекс при старте FastAPI Foundry сервера
        ↓
  search(query) → Релевантные чанки → Контекст для ИИ-модели
```

## 📂 Основные компоненты и файлы

*   **`src/rag/rag_system.py`**: Содержит основной класс `RAGSystem`, который отвечает за загрузку индекса, выполнение векторного поиска и форматирование контекста для ИИ. Глобальный экземпляр `rag_system` доступен для импорта.
*   **`src/rag/indexer.py` (или `rag_indexer.py` в корне)**: Скрипт-индексатор, используемый для создания и обновления индекса RAG. Он обрабатывает исходные документы, разбивает их на чанки, генерирует эмбеддинги и сохраняет их в виде FAISS индекса и метаданных.
*   **`rag_index/`**: Директория для хранения файлов индекса RAG (`faiss.index`, `chunks.json`, `index_info.json`).

## 🧠 Класс `RAGSystem`

`RAGSystem` — это центральный компонент RAG системы. Он инициализируется при запуске FastAPI Foundry и предоставляет методы для управления индексом и выполнения поиска.

### Методы `RAGSystem`

#### `await initialize() -> bool`

Асинхронно загружает индекс FAISS и модель эмбеддингов. Вызывается как часть жизненного цикла приложения FastAPI.

**Returns:** `bool` — `True` если индекс загружен успешно, `False` если RAG недоступен или отключён в конфиге.

---

#### `await search(query, top_k) -> List[Dict]`

Выполняет поиск наиболее релевантных чанков по заданному запросу.

**Args:**

| Параметр | Тип | По умолчанию | Описание |
|---|---|---|---|
| `query` | `str` | — | Текст поискового запроса |
| `top_k` | `int` | `5` | Количество возвращаемых результатов |

**Returns:** `List[Dict]` — список чанков с добавленным полем `score`. Пустой список если индекс не загружен.

---

#### `format_context(results) -> str`

Форматирует результаты поиска в единую строку контекста для промпта.

**Args:**

| Параметр | Тип | Описание |
|---|---|---|
| `results` | `List[Dict]` | Список чанков из `search()` |

**Returns:** `str` — блок контекста с разделителями `=== PROJECT CONTEXT START/END ===`. Возвращает `'No relevant context found.'` если список пуст.

---

#### `await get_status() -> Dict`

Возвращает текущий статус RAG системы.

**Returns:** `dict` — `available`, `enabled`, `loaded`, `index_dir`, `model`, `chunks_count`, `vectors_count`.

---

#### `await reload_index(index_dir) -> bool`

Перезагружает индекс RAG, опционально из нового пути.

**Args:**

| Параметр | Тип | По умолчанию | Описание |
|---|---|---|---|
| `index_dir` | `str \| None` | `None` | Новый путь к индексу. `None` = использовать `config.dir_rag` |

**Returns:** `bool` — `True` если перезагрузка прошла успешно.

---

#### `await clear_index() -> bool`

Удаляет файлы индекса (`faiss.index`, `chunks.json`, `index_info.json`) и сбрасывает состояние.

**Returns:** `bool` — `True` если все файлы удалены, `False` при ошибке доступа к файлу.

---

#### `await get_all_chunks() -> List[Dict]`

Возвращает все чанки с метаданными индекса.

**Returns:** `List[Dict]` — каждый чанк расширен полями `chunk_id` (int) и `text_length` (int). Пустой список если индекс не загружен.

### Формат результата поиска

Результат метода `search()` представляет собой список словарей, каждый из которых описывает найденный чанк:

```python
[
    {
        "source": "installation.md",   # Имя файла-источника
        "section": "Быстрый старт",    # Заголовок секции, из которой взят чанк
        "text": "...",                  # Содержание текстового чанка
        "score": 0.91                  # Косинусное сходство (от 0 до 1), показывающее релевантность
    },
    ...
]
```

## ⚙️ Конфигурация RAG

Настройки системы RAG управляются через файл `config.json` в секции `rag_system`:

```json
{
  "rag_system": {
    "enabled": true,
    "index_dir": "rag_index",
    "model": "sentence-transformers/all-mpnet-base-v2",
    "chunk_size": 1000,
    "top_k": 5
  }
}
```

*   `enabled`: `true` для включения RAG, `false` для отключения. Если `false`, `RAGSystem` не будет загружать индекс.
*   `index_dir`: Путь к директории, где хранятся файлы индекса RAG.
*   `model`: Название модели эмбеддингов (например, `sentence-transformers/all-MiniLM-L6-v2`). **Критически важно, чтобы эта модель совпадала с той, которая использовалась при построении индекса.**
*   `chunk_size`: Максимальный размер текстового фрагмента (чанка) в символах. Должен совпадать с параметром `--chunk-size`, использованным при индексации.
*   `top_k`: Количество наиболее релевантных чанков, которые будут извлечены при каждом запросе поиска по умолчанию.

## 🏗️ Построение индекса RAG

Перед первым использованием RAG необходимо построить векторный индекс, проиндексировав ваши документы.

```powershell
# Запуск индексатора для директории docs/ с сохранением индекса в rag_index/
python rag_indexer.py --docs-dir docs --output-dir rag_index

# Пример с явным указанием параметров, которые должны соответствовать config.json
python rag_indexer.py `
  --docs-dir docs `
  --output-dir rag_index `
  --model sentence-transformers/all-mpnet-base-v2 `
  --chunk-size 1000 `
  --overlap 50
```

> ⚠️ **Важное замечание**: Модель эмбеддингов (`--model`) и размер чанка (`--chunk-size`), используемые при построении индекса, **должны точно совпадать** с соответствующими настройками в `config.json`. Несоответствие этих параметров приведет к некорректным результатам поиска.

## 🔌 API Эндпоинты RAG

Эндпоинты для взаимодействия с системой RAG определены в `src/api/endpoints/rag.py` и доступны по префиксу `/api/v1/rag/`.

*   **`GET /api/v1/rag/status`**: Возвращает текущий статус RAG системы.
*   **`PUT /api/v1/rag/config`**: Позволяет динамически обновлять настройки RAG в `config.json`.
*   **`POST /api/v1/rag/clear`**: Удаляет все файлы индекса и очищает состояние RAG системы.
*   **`POST /api/v1/rag/search`**: (Заглушка) Предназначен для выполнения поиска по запросу. В текущей реализации реальный поиск выполняется непосредственно из эндпоинтов генерации, а этот эндпоинт возвращает mock-данные.
*   **`POST /api/v1/rag/rebuild`**: (Заглушка) Предназначен для запуска перестроения индекса. В текущей реализации возвращает mock-данные.

## 📦 Зависимости

Для работы системы RAG требуются следующие Python-пакеты:

*   `sentence-transformers>=2.2.0`
*   `faiss-cpu>=1.7.0` (или `faiss-gpu` для поддержки GPU)
*   `numpy>=1.24.0`
*   `torch>=2.0.0` (необходим для `sentence-transformers`)

Эти зависимости устанавливаются автоматически при запуске `install.ps1` (если не пропущены) или вручную:

```powershell
pip install sentence-transformers faiss-cpu numpy torch
```

Если необходимые зависимости отсутствуют, `RAGSystem` будет логировать предупреждение и деактивируется, позволяя остальной части сервера FastAPI Foundry продолжать работу без функций RAG.
