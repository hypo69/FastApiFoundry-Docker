    🎓 Дообучение моделей (Fine-Tuning)

    AI Assistant включает встроенную систему дообучения локальных моделей на ваших данных. Это ключевая возможность для построения HelpDesk, AI-секретаря или любого корпоративного ассистента, который должен отвечать именно так, как принято в вашей компании.

    Зачем дообучать модель

    Базовая модель (Mistral, Qwen, Llama и др.) обучена на общих данных интернета. Она не знает:

    • ваших внутренних процессов и регламентов
    • специфической терминологии вашей отрасли
    • правильного тона и стиля общения с вашими клиентами
    • конкретных ответов на типовые вопросы вашей поддержки

    RAG решает часть проблемы — подкладывает нужный контекст в запрос. Fine-tuning идёт дальше — модель усваивает правильное поведение и воспроизводит его без подсказок, быстрее и точнее.

    Архитектура системы дообучения

    Источники данных
  ├─ HelpDesk диалоги (Telegram бот)
  ├─ Ручной ввод через API
  └─ Импорт из CSV / JSONL

        │
        ▼
  DatasetStore (qa_pairs.jsonl)
  ├─ question        — вопрос пользователя
  ├─ answer_correct  — правильный ответ
  ├─ answer_wrong    — пример плохого ответа
  ├─ status          — pending / approved / rejected
  └─ source          — helpdesk / manual / synthetic

        │  (только approved)
        ▼
  FineTuner
  ├─ SFT  — учим на правильных ответах
  ├─ DPO  — учим предпочитать правильный над неправильным
  └─ LoRA — экономичное дообучение адаптером

        │
        ▼
  Дообученная модель / LoRA-адаптер
  └─ подключается как hf::my-model или llama::my-model.gguf

    Формат датасета: QA-пара

    Основная единица данных — QA-пара (question-answer pair):

    Поле Тип Описание
    id UUID Уникальный идентификатор
    question str Вопрос пользователя
    answer_correct str Правильный / идеальный ответ
    answer_wrong str Пример плохого ответа (для DPO)
    status enum pending / approved / rejected
    source str Источник: helpdesk, manual, synthetic
    created_at ISO-8601 Время создания
    metadata dict Произвольные поля (модель, оценка и т.д.)

    Пример пары

    {
  "id": "3f2a1b4c-...",
  "question": "Как сбросить пароль от личного кабинета?",
  "answer_correct": "Перейдите на страницу входа, нажмите «Забыли пароль?» и введите email. Письмо придёт в течение 5 минут.",
  "answer_wrong": "Не знаю, обратитесь в поддержку.",
  "status": "approved",
  "source": "helpdesk",
  "created_at": "2025-12-09T14:30:00",
  "metadata": {"ticket_id": "HD-1042"}
}

    Стратегии дообучения

    SFT — Supervised Fine-Tuning

    Самый простой метод. Модель учится воспроизводить правильные ответы.

    • Нужно: только question + answer_correct
    • Когда: у вас есть база правильных ответов, но нет примеров плохих
    • Результат: модель начинает отвечать в нужном стиле и тоне

    DPO — Direct Preference Optimization

    Модель учится предпочитать правильный ответ над неправильным. Более мощный метод — модель не просто запоминает, а понимает что хорошо, а что плохо.

    • Нужно: question + answer_correct + answer_wrong
    • Когда: у вас есть примеры типичных ошибок (слишком общий ответ, неверный тон, неправильная информация)
    • Результат: модель активно избегает плохих паттернов

    LoRA — Low-Rank Adaptation

    Параметрически эффективное дообучение. Вместо изменения всех весов модели обучается небольшой адаптер (~10–100 MB).

    • Нужно: GPU с 8+ GB VRAM (или Google Colab)
    • Когда: нужно дообучить большую модель (7B+) без полного fine-tuning
    • Результат: адаптер подключается поверх базовой модели, не изменяя её
    Стратегия answer_wrong нужен VRAM Качество Скорость обучения
    SFT нет 8+ GB хорошее быстро
    DPO да 16+ GB отличное медленно
    LoRA нет 8+ GB хорошее быстро

    API Reference

    Базовый URL: /api/v1/training

    Управление датасетом

    Метод Путь Описание
    POST /training/pairs Добавить QA-пару
    GET /training/pairs Список пар (фильтр по status/source)
    GET /training/pairs/{id} Получить пару по ID
    PATCH /training/pairs/{id}/approve Одобрить пару
    PATCH /training/pairs/{id}/reject Отклонить пару
    DELETE /training/pairs/{id} Удалить пару
    GET /training/stats Статистика датасета
    GET /training/export Экспорт одобренных пар

    Запуск обучения

    Метод Путь Описание
    POST /training/run Запустить дообучение

    Примеры запросов

    Добавить пару вручную

    curl -X POST http://localhost:9696/api/v1/training/pairs \
  -H "Content-Type: application/json" \
  -d '{
    "question": "Каков срок доставки?",
    "answer_correct": "Стандартная доставка — 3–5 рабочих дней. Экспресс — 1 день.",
    "answer_wrong": "Зависит от многих факторов.",
    "source": "manual"
  }'

    Одобрить пару

    curl -X PATCH http://localhost:9696/api/v1/training/pairs/<id>/approve

    Посмотреть статистику

    curl http://localhost:9696/api/v1/training/stats
# {"success": true, "stats": {"total": 142, "pending": 30, "approved": 100, "rejected": 12}}

    Запустить DPO-обучение

    curl -X POST http://localhost:9696/api/v1/training/run \
  -H "Content-Type: application/json" \
  -d '{
    "strategy": "dpo",
    "model_id": "hf::mistralai/Mistral-7B-Instruct-v0.3",
    "output_dir": "training_output/helpdesk-v1"
  }'

    Workflow: от диалога до дообученной модели

    1. Накопить диалоги
   └─ HelpDesk бот автоматически сохраняет диалоги
      или добавить вручную через POST /training/pairs

2. Разметить датасет
   └─ GET /training/pairs?status=pending
      PATCH /training/pairs/{id}/approve  — хороший ответ
      PATCH /training/pairs/{id}/reject   — плохой пример

3. Проверить готовность
   └─ GET /training/stats
      → approved >= 100 (минимум для SFT)
      → approved >= 500 (рекомендуется для DPO)

4. Запустить обучение
   └─ POST /training/run {"strategy": "sft", "model_id": "hf::..."}

5. Подключить дообученную модель
   └─ Скопировать веса в ~/.models/
      Указать в config.json: "default_model": "hf::my-helpdesk-model"

    Хранение данных

    Датасет хранится в файле training_data/qa_pairs.jsonl в корне проекта. Формат — JSONL (одна JSON-строка на пару).

    training_data/
├── qa_pairs.jsonl      # Все QA-пары
└── ...                 # Будущие артефакты: веса, логи обучения

    Резервное копирование

    Перед запуском обучения сделайте копию qa_pairs.jsonl — это ваш главный актив.

    Статус реализации

    Хранилище датасета и API полностью работают. Стратегии обучения (SFT, DPO, LoRA) — в разработке. POST /training/run принимает запрос, валидирует данные и возвращает подтверждение — фактический запуск обучения будет добавлен в следующих версиях.

    Минимальный датасет

    Стратегия Минимум пар Рекомендуется
    SFT 50 500+
    DPO 100 (с answer_wrong) 1 000+
    LoRA 50 500+

    Качество дообучения напрямую зависит от качества разметки. Лучше 100 хорошо размеченных пар, чем 1000 сырых.
