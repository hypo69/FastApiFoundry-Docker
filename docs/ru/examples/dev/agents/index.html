    Агенты ИИ в FastAPI Foundry

    Проект: AI Assistant Автор: hypo69 Copyright: © 2024 - 2026 hypo69 Лицензия: MIT

    Модуль агентов в FastAPI Foundry предоставляет расширяемую архитектуру для создания ИИ-агентов, способных получать сообщения от пользователя, принимать решения о вызове внешних инструментов, выполнять эти инструменты и формировать финальный ответ. Это позволяет ИИ-моделям выполнять более сложные, многошаговые задачи.

    🗺️ Архитектура Агентов

    Архитектура агента следует паттерну, в котором ИИ-модель используется как "мозг", принимающий решения о вызове инструментов:

    Пользователь
    ↓ сообщение
BaseAgent.run()                   ← (Основной цикл агента)
    ↓
Сообщения + Описание Инструментов → Foundry (или другой OpenAI-совместимый API)
    ↓
Модель ИИ возвращает tool_calls    ← (Решение модели вызвать инструмент)
    ↓
_execute_tool(name, arguments)    ← (Метод выполнения инструмента агентом)
    ↓
MCP сервер (STDIO / HTTP)         ← (Внешний сервис, выполняющий реальное действие)
    ↓ результат выполнения
Модель ИИ формирует финальный ответ ← (На основе результатов инструментов)
    ↓
Пользователь

    📂 Основные компоненты и файлы

    • src/agents/base.py: Определяет базовый класс BaseAgent, который реализует общий агентный цикл и предоставляет абстрактные методы, которые должны быть реализованы подклассами (например, _execute_tool).
    • src/agents/powershell_agent.py: Пример реализации агента (PowerShellAgent), который использует инструменты для взаимодействия с локальной системой через PowerShell.
    • ToolDefinition: Вспомогательный класс для описания инструментов в формате, совместимом с OpenAI function calling.
    • ToolCallResult: Вспомогательный класс для хранения результатов вызова инструмента.
    • AgentResult: Вспомогательный класс для инкапсуляции конечного результата работы агента (успех, ответ, вызовы инструментов, ошибки).

    💡 Базовый класс BaseAgent

    Класс BaseAgent в base.py предоставляет основу для всех агентов. Для создания нового агента необходимо унаследовать от BaseAgent и реализовать два основных элемента:

    1. Свойство tools: Должно возвращать список объектов ToolDefinition, описывающих инструменты, которые может использовать агент.
    2. Асинхронный метод _execute_tool(self, name: str, arguments: dict): Отвечает за фактическое выполнение выбранного инструмента.

    Методы BaseAgent

    await run(user_message, model, temperature, max_tokens, max_iterations) -> AgentResult

    Запускает агентный цикл: модель → tool_calls → выполнение → модель.

    Args:

    Параметр Тип По умолчанию Описание
    user_message str — Сообщение пользователя
    model str — ID модели Foundry
    temperature float 0.7 Температура генерации
    max_tokens int 2048 Максимум токенов на ответ модели
    max_iterations int 5 Максимум итераций tool-call цикла

    Returns: AgentResult — датакласс с полями:

    Поле Тип Описание
    success bool Успешность выполнения
    answer str Финальный ответ модели
    tool_calls List[ToolCallResult] Журнал вызовов инструментов
    iterations int Количество выполненных итераций
    error str Описание ошибки
    note str Дополнительная информация (например, если модель не поддерживает function calling)

    _execute_tool(name, arguments) -> str — абстрактный

    Выполняет инструмент и возвращает результат как строку.

    Args:

    Параметр Тип Описание
    name str Имя инструмента из tools
    arguments Dict[str, Any] Разобранные JSON-аргументы из tool_call модели

    Returns: str — результат выполнения, передаётся обратно в модель.

    tools_openai() -> List[Dict]

    Преобразует описания инструментов в формат OpenAI function calling.

    Returns: List[Dict] — список словарей {type, function: {name, description, parameters}}.

    Пример структуры нового агента:

    from src.agents.base import BaseAgent, ToolDefinition
from typing import List, Dict, Any

class MyAgent(BaseAgent):
    name: str = "my_agent"  # Уникальное имя агента
    description: str = "Краткое описание функциональности моего агента"

    @property
    def tools(self) -> List[ToolDefinition]:
        return [
            ToolDefinition(
                name="my_tool",
                description="Что делает мой инструмент: принимает строковый ввод и возвращает результат.",
                parameters={
                    "type": "object",
                    "properties": {
                        "input_data": {"type": "string", "description": "Входные данные для инструмента"}
                    },
                    "required": ["input_data"]
                }
            )
        ]

    async def _execute_tool(self, name: str, arguments: Dict[str, Any]) -> str:
        if name == "my_tool":
            # Здесь располагается логика выполнения инструмента
            input_value = arguments.get("input_data", "")
            # ... выполнить действие ...
            return f"Результат выполнения my_tool с вводом: {input_value}"
        return f"❌ Неизвестный инструмент: {name}"

    🎯 Существующие Агенты

    PowerShellAgent

    • Файл: src/agents/powershell_agent.py
    • Имя: powershell
    • Описание: Агент для выполнения команд в локальной системе через PowerShell. Взаимодействует с MCP серверами для изоляции и безопасного выполнения.

    • Инструменты:

      • run_powershell: Выполняет произвольную команду PowerShell (через McpSTDIOServer.ps1).
      • run_wp_cli: Выполняет команды WP-CLI (через McpWPCLIServer.ps1).
      • http_get: Выполняет HTTP GET запрос (прямой вызов).

      Примеры использования (в промпте к агенту): * "Покажи содержимое текущей директории." * "Сколько файлов .py находится в проекте?" * "Список плагинов WordPress."

    RagAgent

    • Файл: src/agents/rag_agent.py
    • Имя: rag
    • Описание: Агент для ответов на вопросы с использованием RAG-контекста. Ищет релевантные фрагменты в FAISS-индексе и формирует ответ на их основе.

    • Инструменты:

      • rag_search: Поиск релевантных фрагментов в базе знаний по запросу (векторный поиск через FAISS).
      • generate_answer: Форматирует финальный промпт с вопросом и найденным контекстом для генерации ответа моделью.

      Примеры использования (в промпте к агенту): * "Что такое RAG и как он работает?" * "Найди информацию о настройке llama.cpp."

      ⚠️ Требует загруженного RAG-индекса. Загрузите документы через POST /api/v1/rag/index перед использованием.

    KeystrokeAgent

    • Файл: src/agents/keystroke_agent.py
    • Имя: keystroke
    • Описание: Агент идентификации пользователей по паттернам нажатий клавиш (клавиатурный почерк). Использует RandomForestClassifier из scikit-learn.

    • Инструменты:

      • train_model: Обучить модель на данных нескольких пользователей. Принимает словарь {username: [[сессия1], [сессия2], ...]}, где каждая сессия — список длительностей нажатий клавиш в мс.
      • predict_user: Идентифицировать пользователя по новой сессии нажатий. Требует предварительного обучения.
      • get_model_info: Получить статус модели: обучена ли, точность, список пользователей.

      Примеры использования (в промпте к агенту): * "Обучи модель на данных пользователей alice и bob." * "Определи пользователя по сессии нажатий [95, 110, 88, 102]." * "Покажи статус обученной модели."

      ⚠️ Требует установленного scikit-learn: pip install scikit-learn.

    ➕ Добавление нового Агента

    Для создания и интеграции нового агента в FastAPI Foundry выполните следующие шаги:

    1. Создайте файл агента: В директории src/agents/ создайте новый файл (например, my_agent.py), унаследуйте класс от BaseAgent и реализуйте свойства name, description и метод _execute_tool (см. пример выше).

    2. Зарегистрируйте агента: Откройте файл src/api/endpoints/agent.py и добавьте ваш новый агент в функцию _build_registry():

      # src/api/endpoints/agent.py

from ...agents.my_agent import MyAgent  # Импортируйте ваш новый агент

def _build_registry():
    return {
        "powershell": PowerShellAgent(foundry_client),
        "rag":        RagAgent(foundry_client),
        "keystroke":  KeystrokeAgent(foundry_client),
        "my_agent":   MyAgent(foundry_client),   # ← Добавьте ваш агент здесь
    }

    3. Используйте агента: После регистрации ваш новый агент станет доступен через API и в веб-интерфейсе (если поддерживается).

      • API вызов: POST /api/v1/agent/run с телом запроса, указывающим "agent": "my_agent".
      • Просмотр списка агентов: GET /api/v1/agent/list.
      • Просмотр инструментов агента: GET /api/v1/agent/my_agent/tools.

    🔌 API Эндпоинты Агентов

    Эндпоинты для управления агентами и взаимодействия с ними расположены в src/api/endpoints/agent.py и доступны по префиксу /api/v1/agent/.

    • GET /api/v1/agent/list: Возвращает список всех зарегистрированных агентов с их именами и описаниями.
    • GET /api/v1/agent/{name}/tools: Возвращает список инструментов, доступных для конкретного агента, указанного по {name}.

    • POST /api/v1/agent/run: Запускает выбранного агента для выполнения задачи.

      Тело запроса для POST /api/v1/agent/run: 

      {
  "message": "покажи содержимое текущей директории",
  "agent": "powershell",
  "model": "qwen3-0.6b-generic-cpu:4:4",
  "temperature": 0.3,
  "max_tokens": 2048,
  "max_iterations": 5
}
      * message: Сообщение или запрос для агента. * agent: Имя агента для использования (например, "powershell"). * model: Имя ИИ-модели, которую агент будет использовать для принятия решений и генерации ответов. Модель должна поддерживать функцию вызова инструментов (function calling), например, Qwen, DeepSeek, Mistral. * temperature, max_tokens, max_iterations: Дополнительные параметры генерации и выполнения агента.

    WindowsOsAgent

    • Файл: src/agents/windows_os_agent.py
    • Имя: windows_os
    • Описание: Специалист по тонким настройкам Windows OS. Реализует сценарий: prompt → RAG (база знаний) → model → MCP tools (реальные данные системы) → ответ.
    • MCP сервер: mcp/src/servers/windows_os_mcp.py (типизированные инструменты, без сырого PS кода)

    • Инструменты:

      Инструмент Описание
      rag_search Поиск в базе знаний по Windows (документация, best practices)
      get_processes Список процессов (топ N по CPU или памяти)
      get_services Службы Windows с фильтром по статусу
      get_disk_info Использование дискового пространства
      get_network_stats Активные TCP соединения и сетевые адаптеры
      get_system_info ОС, CPU, RAM, uptime
      get_startup_items Программы в автозагрузке
      kill_process Завершить процесс по PID

    Пример запроса:

    POST /api/v1/agent/run
{
  "message": "Дай информацию по процессам",
  "agent": "windows_os",
  "model": "qwen3-0.6b-generic-cpu:4:4"
}

    Сценарий выполнения:

    user: "Дай информацию по процессам"
  → rag_search("процессы Windows диагностика")
  → get_processes(sort_by="cpu", top=20)
  → model формирует ответ с реальными данными + рекомендациями
  → ответ пользователю

    ⚠️ Требует загруженного RAG-индекса для полноценной работы. Без индекса агент работает только с реальными данными системы.

    GoogleAgent

    • Файл: src/agents/google_agent.py
    • Имя: google
    • Описание: Агент для работы с Google Workspace — Gmail, Google Calendar, Google Sheets, Google Docs. Авторизация через OAuth2, данные не покидают машину.

    • Инструменты:

      Инструмент Сервис Описание
      gmail_list Gmail Список писем с поддержкой поиска (is:unread, from:...)
      gmail_read Gmail Прочитать письмо по ID
      gmail_send Gmail Отправить письмо
      calendar_list Calendar Предстоящие события
      calendar_create Calendar Создать событие (с участниками)
      sheets_read Sheets Прочитать диапазон ячеек
      sheets_write Sheets Записать данные в таблицу
      docs_read Docs Прочитать текст документа
      docs_append Docs Добавить текст в конец документа

      Примеры использования (в промпте к агенту): * "Покажи мои непрочитанные письма." * "Создай встречу 'Планёрка' завтра с 10:00 до 11:00." * "Прочитай данные из таблицы <spreadsheet_id> диапазон Sheet1!A1:D10." * "Добавь в документ <document_id> текст с итогами встречи."

      ⚠️ Требует настройки OAuth2. Подробная инструкция: Google Agent. Зависимости: pip install -r requirements-google.txt

    QAAgent

    • Файл: src/agents/qa_agent.py
    • Имя: qa
    • Описание: Агент запуска тестов, анализа результатов и формирования отчёта о качестве кода.
    • Сценарий: prompt → run_tests → parse output → summary → answer

    • Инструменты:

      Инструмент Описание
      run_tests Запустить pytest для указанного пути или фильтра, вернуть сводку
      list_test_files Список всех тестовых файлов в tests/
      get_coverage Запустить pytest с --cov и вернуть отчёт покрытия

      Примеры использования: * "Запусти все тесты и покажи результат." * "Запусти только тесты для chat_endpoints." * "Покажи покрытие кода для src/."

    ⏱️ Вотчер тестов (scripts/watch_tests.ps1)

    Автоматически запускает связанные тесты при изменении любого *.py файла в src/.

    # Следить за src/ (по умолчанию)
powershell -ExecutionPolicy Bypass -File .\scripts\watch_tests.ps1

# Следить за конкретной папкой
powershell -ExecutionPolicy Bypass -File .\scripts\watch_tests.ps1 -WatchPath src/api/endpoints

# Запускать все тесты при любом изменении
powershell -ExecutionPolicy Bypass -File .\scripts\watch_tests.ps1 -All
    Параметр По умолчанию Описание
    -WatchPath src Директория для наблюдения
    -All false Запускать все tests/ при любом изменении
    -DebounceMs 1500 Задержка между срабатываниями (ms)

    При изменении chat_endpoints.py вотчер автоматически ищет и запускает tests/unit/test_chat_endpoints.py. Если связанный тест не найден — запускается весь tests/.

    ⚠️ Требования

    • PowerShell 7+: Требуется для корректной работы PowerShellAgent.
    • Foundry Local (или совместимый API): Должен быть запущен и доступен, так как агенты используют его для вызова инструментов на основе решений ИИ-модели.
    • Модель с функцией вызова инструментов: Используемая ИИ-модель (например, Qwen, DeepSeek, Mistral) должна поддерживать функцию tool_calling или function_calling для эффективной работы агентов.
