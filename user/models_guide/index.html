    Руководство по работе с моделями ИИ

    FastAPI Foundry поддерживает четыре бэкенда для запуска моделей ИИ. Каждый доступен через единый REST API и веб-интерфейс.

    Бэкенд Префикс модели Формат Рекомендуется для
    Microsoft Foundry Local (без префикса) ONNX Qwen, DeepSeek, Mistral — быстрый CPU inference
    HuggingFace Transformers hf::model-id PyTorch / safetensors Любые модели с Hub
    llama.cpp llama::model-name GGUF Большие модели на CPU/GPU
    Ollama (настраивается через URL) GGUF Удобное управление моделями

    🤖 Microsoft Foundry Local

    Рекомендуемый бэкенд для моделей Qwen, DeepSeek, Mistral в формате ONNX. FastAPI Foundry автоматически обнаруживает запущенный Foundry Local.

    Запуск сервиса

    foundry service start

    Загрузка модели

    # Скачать модель
foundry model download qwen2.5-0.5b-instruct-generic-cpu

# Список доступных моделей
foundry model list-available

# Загрузить в память
foundry model load qwen2.5-0.5b-instruct-generic-cpu

    Автоопределение порта

    run.py автоматически находит Foundry по процессу Inference.Service.Agent* и сканирует порты [62171, 50477, 58130]. Можно задать явно:

    # .env
FOUNDRY_BASE_URL=http://localhost:50477/v1/

    Автозагрузка модели при старте

    // config.json
{
  "foundry_ai": {
    "auto_load_default": true,
    "default_model": "qwen2.5-0.5b-instruct-generic-cpu"
  }
}

    API

    # Список загруженных моделей Foundry
GET /api/v1/foundry/models/loaded

# Загрузить модель
POST /api/v1/foundry/models/load
{"model_id": "qwen2.5-0.5b-instruct-generic-cpu"}

# Выгрузить модель
POST /api/v1/foundry/models/unload
{"model_id": "qwen2.5-0.5b-instruct-generic-cpu"}

    🤗 HuggingFace Transformers

    Полноценная интеграция с HuggingFace Hub: скачивание, загрузка в память, inference через transformers.pipeline. Реализовано в src/models/hf_client.py.

    Концепция

    HuggingFace Hub
      ↓  snapshot_download()
~/.cache/huggingface/hub/<author>--<model-name>/   ← directories.hf_models (config.json)
      ↓  AutoModelForCausalLM
RAM / VRAM (_loaded_models dict)
      ↓  pipeline("text-generation")
/api/v1/hf/generate

    Модели также сканируются из любой другой директории, если directories.hf_models отличается от дефолта.

    Настройка

    # .env — только токен
HF_TOKEN=hf_ваш_токен   # обязателен для закрытых моделей (Gemma, Llama)

    Директория для скачивания моделей задаётся в config.json:

    {
  "directories": {
    "hf_models": "~/.hf_models"
  }
}

    Куда HuggingFace скачивает модели по умолчанию

    Библиотека huggingface_hub без настройки скачивает модели в стандартный кэш:

    ОС Путь по умолчанию
    Windows C:\Users\<user>\.cache\huggingface\hub
    Linux / macOS ~/.cache/huggingface/hub

    FastAPI Foundry использует тот же путь по умолчанию (~/.cache/huggingface/hub). Если задать directories.hf_models в config.json — модели будут скачиваться туда. Стандартный кэш сканируется автоматически, если hf_models отличается от него.

    Tip

    Изменить директорию можно через веб-интерфейс: Settings → секция Directories → поле HF Models.

    Токен создаётся на huggingface.co/settings/tokens с ролью Read.

    Популярные модели

    Модель Размер Лицензия
    Qwen/Qwen2.5-0.5B-Instruct ~1 GB Открытая
    Qwen/Qwen2.5-1.5B-Instruct ~3 GB Открытая
    TinyLlama/TinyLlama-1.1B-Chat-v1.0 ~2 GB Открытая
    microsoft/phi-2 ~5 GB Открытая
    deepseek-ai/DeepSeek-R1-Distill-Qwen-1.5B ~3 GB Открытая
    mistralai/Mistral-7B-Instruct-v0.3 ~14 GB Нужно принять
    meta-llama/Llama-3.2-1B-Instruct ~2 GB Нужно принять
    google/gemma-2-2b-it ~5 GB Нужно принять

    Для моделей с пометкой «Нужно принять» — откройте страницу модели на huggingface.co и примите лицензию, затем установите HF_TOKEN.

    API

    # Статус интеграции (версии библиотек, CUDA, путь к моделям)
GET /api/v1/hf/status

# Список скачанных и загруженных моделей
GET /api/v1/hf/models

# Список моделей пользователя + популярные публичные модели
GET /api/v1/hf/hub/models

# Скачать модель с Hub
POST /api/v1/hf/models/download
{"model_id": "Qwen/Qwen2.5-0.5B-Instruct"}

# Для закрытой модели — передать токен явно
POST /api/v1/hf/models/download
{"model_id": "google/gemma-2-2b-it", "token": "hf_..."}

# Загрузить в память (CPU или GPU)
POST /api/v1/hf/models/load
{"model_id": "Qwen/Qwen2.5-0.5B-Instruct", "device": "auto"}

# Выгрузить из памяти (освобождает RAM/VRAM)
POST /api/v1/hf/models/unload
{"model_id": "Qwen/Qwen2.5-0.5B-Instruct"}

# Генерация текста
POST /api/v1/hf/generate
{
  "model_id": "Qwen/Qwen2.5-0.5B-Instruct",
  "prompt": "Объясни, что такое RAG",
  "max_new_tokens": 512,
  "temperature": 0.7
}

    Пример ответа /api/v1/hf/status

    {
  "success": true,
  "transformers": {"available": true, "version": "4.40.0"},
  "huggingface_hub": {"available": true, "version": "0.23.0"},
  "torch": {"available": true, "version": "2.1.0", "cuda": false},
  "hf_token_set": true,
  "models_dir": "~/.hf_models"
}

    Использование в чате

    В веб-интерфейсе HuggingFace модели отображаются с префиксом hf:::

    hf::Qwen/Qwen2.5-0.5B-Instruct
hf::TinyLlama/TinyLlama-1.1B-Chat-v1.0

    Модель должна быть скачана и загружена в память перед использованием в чате. При первом обращении через /api/v1/hf/generate загрузка происходит автоматически.

    Зависимости

    pip install transformers accelerate huggingface_hub torch

    Если зависимости не установлены — /api/v1/hf/status вернёт "available": false, остальные эндпоинты вернут понятное сообщение об ошибке.

    🦙 llama.cpp

    Запуск GGUF моделей на CPU или GPU через llama-server.exe. Бинарники для Windows x64 включены в bin/llama-b8802-bin-win-cpu-x64/.

    Настройка

    # .env
LLAMA_SERVER_PATH=.\bin\llama-b8802-bin-win-cpu-x64\llama-server.exe

    Запуск через скрипт

    .\scripts\llama-start.ps1

    Или вручную:

    .\bin\llama-b8802-bin-win-cpu-x64\llama-server.exe `
  --model ~/.models/qwen2.5-0.5b-q4_k_m.gguf `
  --port 9780 `
  --ctx-size 4096

    API

    # Статус llama.cpp сервера
GET /api/v1/llama/status

# Запустить сервер с моделью
POST /api/v1/llama/start
{"model_path": "~/.models/qwen2.5-0.5b-q4_k_m.gguf"}

# Остановить сервер
POST /api/v1/llama/stop

    В чате и генерации llama.cpp модели используются с префиксом llama::.

    🦙 Ollama

    # Установить и запустить
# https://ollama.com/download

ollama run llama2
ollama run qwen2.5:0.5b

    Настройте FOUNDRY_BASE_URL на порт Ollama (по умолчанию 11434):

    FOUNDRY_BASE_URL=http://127.0.0.1:11434/v1/

    ⚙️ Конфигурация моделей

    // config.json
{
  "foundry_ai": {
    "default_model": "qwen2.5-0.5b-instruct-generic-cpu",
    "auto_load_default": false,
    "temperature": 0.7,
    "max_tokens": 2048
  },
  "directories": {
    "models": "~/.models",
    "hf_models": "~/.hf_models"
  }
}
    Параметр Описание
    default_model Модель по умолчанию в веб-интерфейсе
    auto_load_default Загружать модель при старте (только Foundry)
    temperature Температура генерации (0.0 — детерминировано, 2.0 — максимально случайно)
    max_tokens Максимум токенов в ответе
    directories.models Директория GGUF моделей для llama.cpp
    directories.hf_models Директория HuggingFace моделей (переопределяет HF_MODELS_DIR)

    Имена моделей – техническая спецификация

    Как читать имена LLM-моделей и не гадать при выборе: практический how-to

    Если вы хоть раз открывали репозиторий на Hugging Face, Ollama или LM Studio, то наверняка видели строки вроде: Qwen2.5-Coder-32B-Instruct-GGUF-Q4_K_M или Meta-Llama-3.1-8B-Instruct-AWQ. На первый взгляд это случайный набор букв и цифр. На самом деле – компактная техническая спецификация. В имени зашита архитектура, назначение модели, требования к памяти и даже то, как она будет с вами общаться. Научившись «читать» эти этикетки, вы перестанете скачивать гигабайты вслепую и начнёте собирать стек под конкретную задачу.

    Как устроено имя модели

    Унифицированного стандарта нет, но за последние два года сложилась устойчивая конвенция. Имя обычно собирается по схеме:

    [Семейство/Разработчик] + [Версия] - [Размер] - [Специализация] - [Формат/Упаковка]

    Разберём каждый слой.

    1. Семейство и версия

    Примеры: Llama-3.1, Qwen2.5, Mistral-Nemo, DeepSeek-V3, Gemma-2

    Это указывает на лабораторию и поколение архитектуры. Цифры после названия – не маркетинг, а индикатор улучшений: лучше отфильтрованные данные, исправленные баги внимания, оптимизированный токенизатор. Как правило, 3.1 стабильнее и предсказуемее, чем 3.0. Если видите v2, v3 или 2.5 – это почти всегда шаг вперёд по качеству логики и снижению галлюцинаций.

    2. Размер параметров

    Формат: 2B, 7B, 14B, 32B, 70B (B = миллиарды)

    Условный «объём знаний». Больше параметров → глубже рассуждения, лучше удержание длинных зависимостей, но выше требования к памяти. Важный нюанс: качество не растёт линейно с размером. Аккуратно отобранная 14B-модель часто обходит сырую 30B, потому что важнее не количество нейронов, а качество данных и архитектура.

    3. Специализация (суффиксы назначения)

    Здесь кроется главный подвох. Один и тот же размер может вести себя совершенно по-разному в зависимости от этого маркера:

    Суффикс Что означает Когда брать
    base «Голая» предтренерная модель. Умеет продолжать текст, но не понимает команд. Только если вы сами обучаете/дообучаете или используете её как сырьё для fine-tune.
    instruct / IT / chat Прошла instruction tuning. Понимает задания, следует ограничениям, держит структуру ответа. Для 95% сценариев: диалог, API, ассистенты, генерация по промпту.
    coder / code Дополнительно тренировалась на репозиториях, issue, коммитах. Знает синтаксис, паттерны, стек-трейсы. Для написания, рефакторинга, объяснения кода, написания тестов.
    math / reasoning / R1 Заточена под логику, часто использует скрытые цепочки рассуждений. Для математики, сложных выводов, анализа аргументов.

    Правило: если в имени нет instruct или chat, а вы планируете просто спрашивать модель – готовьтесь писать сложные системные промпты и мириться с «плывущим» форматом ответа.

    4. Архитектурные пометки

    Иногда встречаются уточнения в имени или описании: - MoE (Mixture of Experts) – внутри несколько «экспертов», но на каждый токен активируется лишь часть сети. Даёт качество большой модели при расходе памяти меньшей. Пример: Mixtral-8x7B, Qwen2.5-MoE. - 128k / 256k / Long – растянутое контекстное окно. Критично, если вы скормите модели несколько книг или весь репозиторий.

    5. Формат и квантизация

    Часть имени, которую часто добавляют не авторы, а паковщики сообщество (GGUF, AWQ, EXL2). Это способ сжатия весов для локального запуска.

    Обозначение Что значит Потребление памяти (грубо) Когда использовать
    fp16 / bf16 Оригинальные веса ~2 ГБ на 1B параметров Серверы, когда качество важнее всего
    q8_0 8-бит квантизация ~1 ГБ на 1B Почти без потери качества, если есть запас
    q5_k_m 5-бит ~0.75 ГБ на 1B Баланс скорость/качество
    q4_k_m 4-бит (оптимизированная) ~0.65 ГБ на 1B Золотая середина для ноутбуков/слабых GPU
    q3 / q2 Агрессивное сжатие ~0.5 ГБ на 1B Только для очень лёгких моделей или edge-устройств

    Для локального запуска в 2025–2026 годах q4_k_m или q5_k_m в формате GGUF – де-факто стандарт. Вы теряете ~5–10% «интеллекта», но модель влезает в обычную ОЗУ/видеопамять и работает заметно быстрее.

    Как собирать пазл под свои задачи

    1. Определите домен. Код? Текст? Логика? Документы? Суффикс специализации важнее размера.
    2. Посчитайте память. 1B параметров в q4 ≈ 0.6–0.7 ГБ + оверхед на контекст и ОС. Для 8B в q4 закладывайте 5–6 ГБ, для 14B – 9–10 ГБ. Всегда оставляйте запас под активное окно.
    3. Ищите instruct. Без него модель будет «рычать» в ответ на вопросы.
    4. Проверяйте контекст. Если планируете работать с архивами, ищите явное указание 128k+ или long-context. Модели без этого маркера часто «забывают» начало длинного промпта после 8–16k токенов.

    Рекомендации под конкретные сценарии (стандарты 2025–2026)

    🛠 Кодинг и разработка

    Вам нужна модель, которая понимает не только синтаксис, но и архитектурные паттерны, умеет рефакторить, писать тесты и объяснять ошибки без воды.

    Что брать: - Qwen2.5-Coder-32B-Instruct (или актуальные версии на момент чтения) – лидер по качеству кодогенерации и пониманию сложных фреймворков. - DeepSeek-Coder-V2-Lite-Instruct (16B MoE) – отличный баланс качества и потребления памяти. - Для слабых машин (ноутбук, без GPU): CodeGemma-2-7B-Instruct или Phi-3.5-mini-instruct (3.8B).

    Формат: GGUF, квантизация q4_k_m или q5_k_m. Обязательно наличие суффиксов coder + instruct. base-версии здесь бесполезны.

    📚 Большой архив документов (анализ, поиск, RAG)

    Три критических параметра: длинное контекстное окно (128k+), устойчивость к фактическим галлюцинациям и аккуратная работа с инструкциями по структурированию выводов.

    Что брать: - Qwen2.5-14B-Instruct-128k или 32B – отличная фактологическая точность и стабильность на длинных окнах. - Llama-3.1-8B-Instruct-Long – предсказуемая, хорошо документирована, отлично встраивается в RAG-пайплайны. - Если работаете через векторный поиск + контекстная подгрузка, хватит 8–14B. Главное – явно заявленное длинное окно.

    Чего избегать: coder-версий для текста. Они часто «сворачивают» ответы в псевдокод, игнорируют литературный стиль или переоценивают технические детали в ущерб смыслу.

    ☕ Повседневные задачи (заметки, письма, переводы, планирование, простые вопросы)

    Здесь не нужны гиганты. Важна скорость, адекватный тон, низкая стоимость запуска и отсутствие «заумности».

    Что брать: - Meta-Llama-3.2-3B-Instruct или Gemma-2-2B-IT – работают даже на встроенной графике, занимают 2–4 ГБ в q4, отлично справляются с бытовыми промптами. - Qwen2.5-7B-Instruct – если есть запас памяти, даёт заметно более связные и стилистически гибкие ответы. - Mistral-Nemo-12B-Instruct – если нужен баланс между качеством и скоростью (очень эффективная архитектура для своего размера).

    Формат: q4_k_m. Для этих задач переплачивать памятью за q5/q8 обычно бессмысленно: вы не заметите разницы в переписке или планировании.

    Краткий чек-лист перед скачиванием

    1. Есть ли instruct / IT / chat? → Если нет, не берите для диалога.
    2. Подходит ли суффикс под домен (coder, math, long)? → Специализация > размер.
    3. Влезает ли в память с запасом под контекст? → 1B ≈ 0.65 ГБ в q4 + 20% на окно.
    4. Выбран ли адекватный формат? → GGUF q4_k_m как стартовая точка всегда безопасен.
